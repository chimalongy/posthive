01 Register your app — get static credentials
1
Create a TikTok Developer account
Visit developers.tiktok.com and sign up with a business email.
⌄
  • Go to developers.tiktok.com — this is completely separate from your personal TikTok account.
  • Click Sign up, enter your email, verify with the code sent to your inbox (you have 1 minute), then set a password and accept Terms.
  • Strongly recommended: go to My organizations → Create organization. This represents the owning business entity of your app and is required for production apps serving other users.
  • Individual developer accounts work for sandbox/testing but are not recommended for real integrations.
2
Register your app
Profile icon → Manage apps → Connect an app → select your Organization as owner.
⌄
  • Enter a meaningful App Name — generic names like "App12345" or "Test App" are rejected at review. Use your brand name.
  • Select a Category, upload an App Icon (must meet TikTok icon specs), and write a clear Description of what your app does.
  • Under platforms, enable Configure for Web — this is what enables OAuth redirect URIs for a web app like PostHive.
  • Enter a real HTTPS Website URL. Localhost does not work in production. For local dev, see Step 5 (ngrok).
  • Also required (apps created after September 9, 2024): Terms of Service URL and Privacy Policy URL. These must be real, publicly accessible pages.
⚠TikTok requires your application to be intended for a wide audience and will reject apps built for internal/private use only. Your ToS and Privacy Policy pages must be real, not placeholder pages.
3
Add products to your app
Left panel → + Add products → add Login Kit and Content Posting API.
⌄
  • Login Kit — enables OAuth 2.0 user authentication and token generation. Requires Redirect URI (HTTPS), ToS URL, Privacy Policy URL.
  • Content Posting API — enables programmatic video and photo publishing. Enable the Direct Post option for posts to go live immediately (may require a separate audit).
  • Required OAuth scopes for PostHive: video.upload video.publish user.info.basic
  • Add a Redirect URI in Login Kit config — this is the URL TikTok sends the user back to after they authorize (e.g. https://yourdomain.com/api/tiktok/callback).
  • A maximum of 10 Redirect URIs is supported. URIs must be absolute, begin with https, and must be static (no query parameters).
4
Get your Client Key and Client Secret  Static credentials
Manage apps → Your App → Basic information → Credentials → click the eye icon.
⌄
Credential Where to find it Notes
client_key Developer Portal → Manage apps → Your App → Basic information → Credentials (click eye icon) Static Public identifier. Safe to use in redirect URLs. Does not expire.
client_secret Same location as client_key. Reveal with eye icon. Server only Never send to browser. Never commit to git. Does not expire.
5
Verify your domain  Production only — skip in Sandbox
App page → URL Properties → Verify Properties → choose Domain or URL Prefix.
⌄
  • Click URL Properties at the top of your app page → Verify Properties → choose Domain (verifies entire domain including subdomains) or URL Prefix (specific path).
  • TikTok gives you two options: download a signature file and upload to your server root, or add a DNS TXT record to your domain host.
  • Sandbox apps skip this step entirely — no domain verification is required in sandbox mode. This makes sandbox ideal for early development.
  • For local development using sandbox, use ngrok to expose your local server:
npm install -g ngrok
ngrok http 3000
# Copy the https://xxxxx.ngrok.io URL
# Set it as your Redirect URI in Login Kit config
# Update TIKTOK_REDIRECT_URI in .env.local each session
ℹngrok URLs change every session on the free plan. Use a paid ngrok account with a fixed subdomain, or set up a staging server, to avoid updating the redirect URI each time.
6
Submit for App Review  5–10 business days
App review section → fill use case → submit. Required for real public posting.
⌄
  • Go to the App review section on your app page and fill in a clear description of your use case.
  • You must provide: a demo video of your posting flow, Privacy Policy URL, Terms of Service URL, and a statement of how user data is handled.
  • Approval typically takes 5–10 business days in 2026. TikTok provides no official timeline guarantees — factor this into your launch plan.
  • Until approved: Direct Post (public posting) is unavailable. All posts land in the creator's TikTok inbox as private drafts only.
  • If rejected, you'll see review comments in the history icon on your app page. Address them and resubmit.
⚠Enabling the Direct Post option under Content Posting API may require an additional separate audit beyond the standard app review. Plan accordingly.
02 OAuth 2.0 flow — get access token + refresh token
7
Run the OAuth 2.0 + PKCE authorization flow  Generates tokens
Runs each time a user connects their TikTok account to PostHive. Returns access_token and refresh_token.
⌄
User clicks "Connect TikTok"
→
Redirect to /v2/auth/authorize/
→
User grants permission on TikTok
→
TikTok returns auth code to redirect_uri
→
Server POSTs to /v2/oauth/token/
→
Store access_token + refresh_token

Token exchange request:

POST https://open.tiktokapis.com/v2/oauth/token/
Content-Type: application/x-www-form-urlencoded

client_key=YOUR_CLIENT_KEY
client_secret=YOUR_CLIENT_SECRET
code=AUTH_CODE_FROM_CALLBACK
grant_type=authorization_code
redirect_uri=YOUR_REDIRECT_URI

Successful response — what you receive and what to store:

{
  "access_token":      "act.example12345...",   // ← STORE THIS
  "expires_in":         86400,                   // 24 hours in seconds
  "refresh_token":     "rft.example12345...",   // ← STORE THIS (replaces open_id)
  "refresh_expires_in": 31536000,               // 365 days in seconds
  "open_id":           "afd97af1-b87b-...",    // ← DO NOT STORE — no longer needed for auth
  "scope":             "user.info.basic,video.upload,video.publish",
  "token_type":        "Bearer"
}
24 hrs
access_token TTL
(prefix: act.)
365 days
refresh_token TTL
(prefix: rft.)
⚠Both tokens rotate on every refresh call. After calling /v2/oauth/token/ with grant_type=refresh_token, the old tokens become invalid immediately. Always save the new access_token and refresh_token pair returned in the response.

Token refresh request (call before every post if access_token is expired):

POST https://open.tiktokapis.com/v2/oauth/token/
Content-Type: application/x-www-form-urlencoded

client_key=YOUR_CLIENT_KEY
client_secret=YOUR_CLIENT_SECRET
grant_type=refresh_token
refresh_token=STORED_REFRESH_TOKEN
ℹStore token_expires_at = Date.now() + (expires_in * 1000) alongside the token. Before every post, check if expired. If yes, refresh first. This avoids silent 401 failures mid-post.
03 TikTok Sandbox — test without app review
S1
Enable Sandbox mode on your app
App page → toggle Production → Sandbox → Create Sandbox → name it → clone from production.
⌄
  • On your app page, switch the toggle next to your app's name from Production to Sandbox.
  • Click Create Sandbox → enter a name (e.g. "PostHive Dev") → choose Clone from Production so all your product configs copy over → click Confirm.
  • Add products inside the sandbox (Login Kit, Content Posting API) and click Apply changes.
  • You can create up to 5 sandboxes per app. Use the [+] button in the left nav to add more.
  • No domain verification required in sandbox — start testing immediately.
  • Sandbox uses the same Client Key and Client Secret as your production app. No new credentials to generate.
✓Once you're ready to go live, you can import your tested sandbox configuration directly into your production Draft: switch to Production → Draft → Import → select your sandbox.
S2
Add test TikTok accounts (target users)
Sandbox settings → Target users → Add account. Up to 10 accounts allowed.
⌄
  • Go to Sandbox settings → Target users → Add account.
  • You'll be redirected to log into a TikTok account. Use a real TikTok account you own (create dedicated test accounts for development — separate from real creator accounts).
  • The account must accept TikTok Developer Terms of Service once during this flow.
  • Only these added accounts can go through the OAuth flow in sandbox mode. All other TikTok accounts will be blocked.
  • You can add up to 10 target user accounts per sandbox.
S3
Sandbox vs Production — what works and what doesn't
Key differences between sandbox and production environments.
⌄
🧪 Sandbox
Full OAuth flow (access_token + refresh_token)
Video/photo uploads to TikTok inbox as drafts
Token refresh flow works identically
Creator info query
Publish status polling
File size up to 128 MB
No domain verification needed
No app review needed — ready instantly
Same Client Key/Secret as production
🚀 Production
Direct Post — videos go live publicly
File size up to 10 GB
Unlimited users (any TikTok account)
Duration up to 10 minutes
Webhook callbacks for upload status
Duet/Stitch permissions
Branded content disclosures
Geo-targeting of video visibility
✕Sandbox does NOT support: Direct Post for public videos, Data Portability API, files over 128 MB, or more than 10 test users.
04 PostHive — implement sandbox + production toggle
✓
Implement sandbox/production mode switching in PostHive
Add an is_sandbox toggle in the TikTok CredentialModal and switch API endpoints accordingly.
⌄
  • Add a Sandbox mode toggle in the TikTok CredentialModal — a clearly labelled switch: "Test mode (Sandbox)" on/off.
  • When sandbox is ON: store is_sandbox: true in credentials and show a yellow banner on the platform card: "Sandbox — posts land in your TikTok inbox as drafts".
  • When sandbox is OFF: use the direct post endpoint for publicly published content.
// lib/platformPosters/tiktok.js
// Choose endpoint based on is_sandbox flag

const ENDPOINT = credentials.is_sandbox
  ? 'https://open.tiktokapis.com/v2/post/publish/inbox/video/init/'
  : 'https://open.tiktokapis.com/v2/post/publish/video/init/';
ℹIn sandbox mode, videos appear in the creator's TikTok inbox as drafts — they are never published publicly. This is perfect for testing your entire posting pipeline end-to-end before going live.
05 Final credential fields — CredentialModal form
✓
Store exactly these fields in connected_platforms.credentials (jsonb)
Updated credential list — open_id removed, refresh_token and is_sandbox added.
⌄
Field Status Where / How to get it Notes
client_key ✓ Current Developer Portal → Manage apps → Basic information → Credentials Static, does not expire
client_secret ✓ Current Same location as client_key. Eye icon to reveal. Server-side only. Never expose to browser or logs
access_token ✓ Current Returned from OAuth token exchange at /v2/oauth/token/ TTL: 24 hrs. Prefix: act. Refresh before every post.
refresh_token Replaces open_id Returned from same OAuth token exchange response TTL: 365 days. Prefix: rft. Both tokens rotate on each refresh.
token_expires_at Add this Calculate: Date.now() + expires_in * 1000 Enables proactive refresh before TTL expires
is_sandbox Add this From CredentialModal toggle set by the user Switches between inbox endpoint and direct post endpoint
open_id Removed — No longer required for API v2 authentication. Runtime metadata only. Do not store.
⚠In your CredentialModal for TikTok, the client_key and client_secret are entered once by the user from their Developer Portal. The access_token and refresh_token are obtained automatically when the user clicks "Authorize with TikTok" — your app handles the OAuth flow server-side and stores the returned tokens. Do not ask users to manually paste tokens.
